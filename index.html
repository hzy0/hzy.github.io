关机过程中死锁问题

问题描述

手机通过OTG连接鼠标、键盘灯设备，概率出现关不了机。

初步排查

手机黑屏、连接电脑没有adb口、按power2~4秒没有开机说明手机没有进入关机状态。Tcard log未能找到有效信息，只能确定上层的关机流程已经跑完了。怀疑内核卡死在关机流程！需要ramdump辅助分析，而此时系统并没有crash,adb也不能用,需要硬件级别触发ramdump。把电源键长按功能定制为进入ramdump。将dtsi以下代码（长安重启）删除！

重新复现问题，问题复现后，连接电脑，打开QFIL Configuration，长按power键，进dump后抓出ram dump！

分析过程

问题定位

拿到ramdump后，使用crash-utility工具解析出来，开始分析。首先，查看8个cpu的就绪队列上只有idle进程。系统空闲，这也符合关机过程的负载情况，但是其他所有进程却都没有拿到CPU资源，比如关机流程这个时候正常的话应该持有cpu资源吧！

查看是不是有可以的D状态进程！比较可疑的几个我们用红色字母标记出来！！特别是这个init进程是首要的分析对象，因为android世界里所有的关机、重启流程都统一由init进程发起底层流程！所以内核的关机流程应该是运行在init进程空间里面！

锁的推导

通常情况下对于变量的推导，可以从当前函数往前或者往后去推导。

往前推导：从device_shutdown函数的C代码可以看到dev和parent是局部变量，我们断定可以下这个函数的栈帧中找到这两个变量。但是这个案例中，看看当前init进程堆栈中device_shutdown函数的详细堆栈情况。函数汇编代码我看起来有点吃力，不知道是不是代码被优化过了，device_shutdown函数的栈帧似乎并没有用来存储本地局部变量！能力有限，放弃往前推导。

往后推导：mutex_lock函数接收一个参数，就是metux。我们看看往后能不能拿到这个锁的地址，进而找到parent这个device！mutex_lock函数中接收的x0就是这个mutex，红色1中x0会赋值给x19，然后红色2调用__ mutex_lock_slowpath函数，由于 __ mutex_lock_slowpath函数需要用到想x19，所以x19会先被保存在__mutex_lock_slowpath的栈帧当中，如红色3所示！

看看__mutex_lock_slowpath函数栈帧偏移16的红框位置中应该就是被保存的x19，也就是mutex！！

通过mutex在struct device结构体中的偏移，我们就可以知道这个parent的地址为多少。parent的地址为ffffffcbf3a3b070 – 0x60 = ffffffcbf3a3b010，看看这个device的关键信息。对应的内核对象名字可以猜测是哪个模块的设备，mutex可以看到当前锁的owner（就是锁的持有者），还有可以看出哪些进程正在等待这个锁！下面我们一一分析！

Android稳定性基础知识

概述

对于Android系统来说，稳定性大体分为Framework/App Crash，Native Crash，以及Kernel Crash。当然也包括一些其他的子系统如adsp/modem/tz等，不过这不是我们学习的范围。

  • 对于framework层或者app层的Crash(即Java层面Crash)，那么往往是通过抛出未捕获异常而导致的Crash。这类crash在一些极限且不影响功能的前提下可以通过捕获异常获取非法判断来规避问题，但是大部分情况下还是要从业务逻辑和流程分析根本原因。这类异常涉及到很多业务逻辑的，通常依赖java的回溯去分析，不需要借助特殊的工具。
  • Native Crash，即C/C++层面的Crash，这是介于系统framework层与Linux层之间的一层，比如jni或者一些native的服务。这类可能需要tombstone/coredump，借助gdb工具来帮忙分析问题。
  • 至于Kernel Crash，很多情况是发生Kernel panic，对于内核崩溃往往是驱动或者硬件出现故障。涉及到内核发生crash时内存取证和类似trace32或者crash-utility等工具。

这里主要整理下分析native和kernel的crash问题一些基础知识。

Native Crash

  • 手段：当native进程发生crash的时候，android的debuggerd守护进程会自动打印对应进程的信息信息到目录/data/tombstones/tombstone_xx文件，或者通过配置生成coredump调试信息。
  • 参考：http://gityuan.com/2016/06/25/android-native-crash/和http://source.android.com/devices/tech/debug/
  • 案例：使用android源码中自带的crasher制作crash，学习分析。我们主要用到coredump和gdb

coredump

  • 通常情况下 coredump 包含了程序运行时的内存，寄存器状态，堆栈指针，内存管理信息等，可以理解为把程序工作的当前状态存储成一个文件。Android 系统是基于Linux 内核，Linux 内核原生就支持这种机制。
  • 产生的原因很多，通常有：内存访问越界、 非法指针、堆栈溢出等。这些异常出现的时候系统会发出相对应的信号使程序退出。而这些信号也会触发生成程序的coredump信息。

ramdump

内核crash的原因是很多样且复杂的，大多数不是通过简单的回溯定位就能分析其根本原因。需要借助以下机制和工具来进行分析，内存取证对复杂的crash问题分析有很大的帮助。内存取证的方法不同的芯片厂商使用的方法不同，这里主要用到高通平台的ramdump。通过高通的定制，开启了相关机制后，在内核crash的时候系统会进入ramdump mode，然后使用其工具将整个内存的信息保存出来。这里不作详细的介绍。

Linux内核slub使用与调试

前言

在Linux中，伙伴系统（buddy system）是以页为单位管理和分配内存。但是现实在使用内存的时候不可能总是需要页的倍数大小的内存块，为了能支持更小粒度的内存非配，内核先后出现了slab/slub/slob三种小内存分配器；slub分配器是slab分配器的优化版本，而slob是一种精简的小内存分配算法，主要用于嵌入式系统。慢慢的slab分配器或许会被slub取代，现在android 的linux内核多是使用了slub。

我们都知道，只要是需要我们自己动态分配释放的，都可能在使用的过程中的疏忽导致一些问题的出现。所以学习内存分配器的原理和调试方法是很有必要的。内存问题大家都已经做了很多的总结，大多是以下几类：

  • 内存泄露（leak），分配使用完之后忘了释放，导致内存占用不断增长；最终可能导致内存紧张、系统异常、甚至OOM
  • 内存越界（overrun），我们通常叫做内存踩踏，访问了分配的区域之外的内存，覆盖了不属于自己的数据；从而导致其他模块异常或系统崩溃
  • 使用已释放的内存（use after free），已释放的内存会在某些时刻被其他需要的模块申请，若释放后还往里面写数据，会导致其他模块的数据被修改而导致问题；
  • 使用未初始化的数据（use uninitialised bytes），缺省模式下alloc分配的内存是不被初始化的，内存值是随机的，直接使用的话可能会造成很严重的后果。
    ….

而slub自带了故障诊断机制，给内存问题的分析带来了很多方便之处。本文的目标就是学习如何诊断slub内存问题。

机制

系统是否使用了slub，如何确认？
  • 可以通过查看/sys/kernel/slab目录是否存在来判断，有就是slub，没有就是slab；
  • 或者查看内核配置宏是否开启了CONFIG_SLUB，若开启此宏便是使用了slub分配器！
slub有哪些调试机制？
  • Red zone来自于橄榄球术语，是指球场底线附近的区域，slub通过在每一个对象后面额外添加一块red zone区域来帮助检测越界(overrun)问题，在red zone里填充了特征字符，如果代码访问到了red zone就意味着越界了。
  • Poisoning是通过往slub对象中填充特征字符的方式来检测use-after-free、use-uninitialised等问题，比如在分配slub对象时填充0x5a，在释放时填充0x6b，然后debug代码检查时如果看到本该是0x6b的位置变成了别的内容，就可能是发生了use-after-free，而本该是0x5a的位置如果变成了其它内容就意味着可能是use-uninitialised问题。相关poison字节定义在kernel/include/linux/poison.h中。
Slub的debug机制有哪些相关的内核宏？
  • CONFIG_SLUB_DEBUG调试机制开关宏，开启此宏并不代表激活了debug机制，还需要配置调试选项！
  • CONFIG_SLUB_DEBUG_ON此宏开机默认激活debug选项，查看默认选项查看DEBUG_DEFAULT_FLAGS.否则需要通过内核cmline添加slub_debug参数制定选项，详见Slub.txt！
  • CONFIG_SLUB_DEBUG_PANIC_ON此宏打开后，当slub检测到错误，会主动触发panic！
slub有哪些调试选项？

参考内核文档kernel/Documentation/vm/slub.txt可知

  • F Sanity checks on (enables SLAB_DEBUG_FREE. Sorry SLAB legacy issues)
  • Z Red zoning
  • P Poisoning (object and padding)
  • U User tracking (free and alloc)
  • T Trace (please only use on single slabs)
  • A Toggle failslab filter mark for the cache
  • O Switch debugging off for caches that would have caused higher minimum slab orders
  • “-” Switch all debugging off (useful if the kernel is configured with CONFIG_SLUB_DEBUG_ON)

framwork全局符号表溢出问题

问题描述

10台机器进行某项自动化测试，一轮5天，发现一台机器没有完成测试就停止了。

分析过程

1.拿到log，可以快速地定位到system_server发生了crash导致android层重启，且直接原因是全局引用表溢出，虚拟机dump信息如下：

由上面的关键信息可以得到，虚拟机全局引用表对象引用个数限制为51200个；最后在创建一个ObserverEntry对象；而引用数最多的两个对象是ObserverEntry和WeakReference（怀疑是binder实体的弱引用，ObserverEntry代表一个内容监听器在framwork的实例，是用来回调到app端的，所以ObserverEntry和WeakReference 数量接近）。这里只能从ObserverEntry对象入手继续分析，结合ContentService的实现，怀疑有应用反复注册了内容监听器而没有在适当的时候释放。需要现场来进行定位确认，但是复现问题的机器现场已经丢失，只能拿做完测试但未复现问题、且没重启过的机器来做现场调试。

2.拿到一台测试后没有重启的机器。既然ObserverEntry的对象这么多，那么就dumpsys content看看ContentService的信息，想进一步了解其内部信息可以看看ContentService的registerContentObserver、unregisterContentObserver和dump函数的逻辑。这里不详细说，直接看结果。这里有大量的监听器，且注册者都是pid=1266的进程。通过ps命令查看或者log，发现pid 1266是SystemUI。

现在我们基本确定是SystemUI反复注册了always_on_display_constants的监听器而没有释放，通过监听器内容搜索SystemUI的代码，找到其中注册always_on_display_constants监听器的逻辑，发现在启动DozeService的时候总共会注册三个always_on_display_constants的监听器，但是没有看到有注销的地方。这应该就是ObserverEntry对象泄露的原因。

3.那DozeService什么时候启动，又是什么时候退出呢？再看看log，有大量的启动停止DozeService的log记录，启动记录7609条，每启动退出一次泄露3个，在能看到的log记录总共就泄露22827个。非常接近前面虚拟机dump出来的引用表信息ObserverEntry的数量。

自动化测试通常都比较耗时，问题不好验证，最好是找到复现问题的步骤。这个问题也比较明显，DozeService会在灭屏的时候启动，亮屏的时候退出，这是framework的逻辑，这里不详细分析。我们通过亮灭屏的动作和dumpsys content看看我们前面的猜测是不是对的。灭屏后亮屏可以看到observer的对象增加3，而增加的3个都是always_on_display_constants

4.和测试的同事确认这个自动化测试用例中是否有亮灭屏的动作，得到了肯定的答复，是客户要求的测试用例！这样我们就有了复现问题的方法，后续修改验证也就方便了许多。

验证方法

由于问题比较清晰，我们直接测试20000次”休眠唤醒”的动作来对比验证。准备两台机器，一台刷老版本；一台刷修改后的版本。验证结果为老版本8000次左右就发生了crash，而修改后的版本20000次后系统依然正常。
