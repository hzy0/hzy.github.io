卡在开机动画界面的一个案例分析

问题描述

设置锁屏密码进行开关机压力测试，低概率出现输入密码后卡在开机动画，无法进入主界面。

分析过程

初步分析

卡在开机动画问题,一般从systemserver入手。通过systemserver关键字，从log很容易就可以确定systemserver主线程卡在StartAudioService中；并且通过adb shell ps命令察看进程状态确认system_server的进程状态一直处于futex_wait_queue_me中。此问题同样需要trace协助定位具体问题，而user版本无法调试，需要userdeubg版本进行复现。

细节分析

userdeubg复现到问题后，多次抓取systemserver的trace进行分析。主线程意料之中的卡在我们初步分析确定的点上：

显然，这不是systemserver自身的问题。大概了解audio子系统的都知道audioservice其实完全依赖audioserver这个进程；我们需要看看audioserver这个服务进程的trace。对于这个问题，我们主要关心systemsever回溯最后是get_audio_policy_service()，在获取audio policy这个binder服务，所以我们先看看服务列表里面audio相关的服务，audio_policy接口描述为空！

也就是说audio_policy未能注册进系统，那我们主要看看audioserver进程的主线程trace。主线程正在注册audio policy服务，但是在启动的时候加载hal层的部件并等待完成。

android8.0的版本之后，framework和hardware分离，各自运行在独立的进程中。所以加载hal模块的实际工作在audio hal service里面完成！可以从上面的回溯看出，audioserver主线程正在通过HIDL向audio hardware请求并等待结果，但是audio hardware一直没有返回。至此，我们还需要进一步查看audio hardware service进程的回溯！直接把audio hardware service 的trace拉出来分析：

通过以上回溯分析，我们整理一下问题的逻辑：
(1). systemserver : StartAudioServiceAudioSystem.setMasterMono ->get_audio_policy_service
(2). Audioserver : AudioPolicyService::publish loadHwModule waitForResponse
(3).audio hardware service : 主线程在adev_open时等待全局锁adev_init_lock；1520线程在adev_close中已经先持有了adev_init_lock，并等待spkr_calibration_thread线程退出才能继续做反初始化；但是此时speaker校准线程spkr_calibration_thread却一直在等待spkr_calib_cancel信号没有退出。
(4). 为什么在android启动的过程中audio hardware service会有反初始化的动作？audioserver会重启hal层服务！而这里framework为什么重启是因为android都开启了FDE，FDE流程在这里就不说了！

猜测此问题应该可以在apkr校准失败的情况下，重启android层必现，试验发现确实如此！找到必现路径可以用于修改方案的验证！这个问题分析过程中也一直跟audio的同时沟通，对于最后的解决方案：第一我们觉得这个校准失败的问题必须解决；同时我这边想了个办法，就是在FDE解密用户分区之前的最小framework启动过程（也就是用户输入密码前）不要启动进行speaker校准，等到解密重新挂在data分区成功，重启framework后的这次启动才进行speaker校准，这样就能避开这个问题。

总结

对于hang问题，我们通常会用到一些工具来辅助分析；比如，kill -3、debugged、addr2line等等。这个例子就是如此，所以掌握这些工具的基本用法是很有必要的。
