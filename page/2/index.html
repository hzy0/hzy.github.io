Android平台上使用coredump+gdb

Android的底层是Linux，所以，对于用户空间的稳定性问题可以使用gdb+coredump来分析。coredump是linux原始支持的一个机制，可以在进程发生严重问题时候dump出问题进程的进程空间，可以通过gdb调试coredump定位问题。

如何恢复java回溯？

关于恢复java堆栈的问题，我这边尝试了以下步骤：

1、修改一下两个wenjian文件
/build/core/dex_preopt_libart.mk “–no-generate-debug-info” to “–generate-debug-info”
/build/core/dex_preopt_libart_boot.mk “–no-generate-debug-info” to “–generate-debug-info”

2、make systemimage

3、关闭vbmeta、重刷systemimage

4、setprop persist.debug.trace 1 ; 然后重启

5、修改system_server进程的coredump_filter为00000027

6、kill -6 pid_of_systemserver

启动gdb加载这个system_server的coredump：

  • core-file 加载coredump文件；
  • file 加载带符号表的可执行文件app_process64
  • set sysroot 设置工作目录
  • info sharedlibrary查看确保so库、oat、odex等加载正常

然后bt f看看system_server主线程完整的回溯栈信息如下，可以看到包含了java部分的回溯。至于java的回溯不像native的部分可以直接得到所在代码的行，可能需要根据加载地址在反编译oat等来定位。

gdb常用命令

1、info sharedlibrary

这个命令会把进程所有的so、oat等全部列出来；可以通过它确认我们少了那些符号表文件

2、set pagination off

这个命令关闭gdb分页显示，不用老是回车显示下一页面

3、info proc可选查看进程信息

4、info threads查看所有线程

5、info info variables查看所有的全局变量

6、切换线程上下文、切换栈帧、查看当前栈帧的局部变量、形参等。

7、p命令，查看变量、内存内容等

8、反汇编
